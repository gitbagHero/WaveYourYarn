> 计划日期：2026-07-21
>
> 开发基线：v0.2.5
>
> 版本范围：v0.3.0、v0.3.1、v0.3.2
>
> 状态：规划完成，关键产品决策已于 2026-07-21 确认
>
> 决策记录：`docs/v0.3-decisions.md`
>

## 1. 阶段目标

v0.3.x 的目标不是简单增加一个“调用大模型”按钮，而是建立一套用户可控、结果可解释、失败可恢复的本地 AI 报告系统：

```text
版本化本地音乐数据
  → 披露预览与明确确认
  → 可取消的 LLM 任务
  → 严格校验的结构化报告
  → 结论与本地证据绑定
  → 本地历史、离线阅读和安全导出
```

版本边界：

| 版本   | 核心交付                                              | 用户可见结果                                     |
| ------ | ----------------------------------------------------- | ------------------------------------------------ |
| v0.3.0 | LLM profile、网络与隐私边界、JobManager、分析数据契约 | 可配置并测试模型连接，可预览将发送的数据         |
| v0.3.1 | 确定性事实、结构化生成、证据链、报告历史              | 可生成、阅读、重命名、删除、重新生成 AI 音乐报告 |
| v0.3.2 | Markdown/HTML 导出、主题与隐私预览、版本比较          | 可把报告保存为可分享且可脱敏的个人成果           |

v0.3.x 明确不做：

- AI 自动创建、修改或删除网易云歌单；
- Apple Music 授权、匹配或迁移；
- 云端账号、报告同步、遥测或托管；
- 把音乐画像描述为心理诊断、健康结论或客观事实；
- 未经用户确认自动上传音乐数据；
- 在日志、数据库公开字段或 renderer 中保存 API Key。

v1.0 的网易云能力保持完全只读。v0.3.x 不预留歌单整理、网易云写入或 Apple Music 的 IPC、数据库表和页面入口。

## 3. 核心设计

### 3.1 分层

```text
Renderer
  ├── LLM 设置与连接测试
  ├── 数据披露预览
  ├── 任务状态与取消
  └── 报告阅读、历史与导出
          ↓ typed preload API
IPC boundary
  ├── runtime validation
  ├── sender validation
  └── IpcResult envelope
          ↓
Main domain
  ├── AnalysisDatasetService
  ├── DataDisclosurePolicy
  ├── LLMProfileService
  ├── JobManager
  ├── AIReportService
  └── ReportExportService
          ↓
Adapters / repositories
  ├── OpenAICompatibleProvider
  ├── ExternalUrlPolicy / NetworkTransport
  ├── SecureStorageService
  └── SQLite repositories
```

硬边界：

- renderer 只能看到 `hasApiKey`，永远拿不到 API Key；
- 网络请求只在 main 发起；
- LLM adapter 不接收网易云 Cookie，也不能访问网易云写 adapter；
- 报告 service 只接收版本化 dataset 和公开 profile；
- 未通过输出 schema 校验的内容不能保存为成功报告；
- AI 任务不具备任何远端音乐账号写权限。

### 3.2 分析数据契约 v1

在接入真实 LLM 前先升级 `MusicAnalysisDataset`：

```ts
interface MusicAnalysisDatasetV1 {
  schemaVersion: 1
  source: StatisticsSourceInfo
  scope: {
    selection: 'most_recent'
    requestedSongLimit: 100
    availableSongCount: number
    includedSongCount: number
    truncated: boolean
    timePrecision: 'source_timestamp' | 'mixed' | 'order_only' | 'none'
  }
  summary: MusicStatsSummary
  compactSongs: Array<{
    ncmSongId: string
    name: string
    artists: string[]
    album?: string
    time?: number
    orderIndex?: number
  }>
  digest: `sha256:${string}`
  generatedAt: string
}
```

规则：

- `ncmSongId` 是 evidence 引用的唯一歌曲标识；
- schema version 只增不改，旧报告保留生成时的版本；
- 首版默认且最多发送最近 100 首，实际发送数量必须由用户在披露预览确认；
- 不发送 Cookie、用户目录、原始 API 响应、未选择歌单或诊断日志；
- 对 liked 时间缺失的数据显式标记为顺序近似，不能伪造精确时间线；
- 生成 dataset digest，用于报告来源审计和重复生成比较。

### 3.3 报告 schema v1

建议固定为业务层 schema，不直接持久化厂商响应：

```ts
interface AIReportContentV1 {
  schemaVersion: 1
  title: string
  subtitle: string
  tasteSnapshot: {
    summary: string
    keywords: string[]
    evidence?: EvidenceRef // prompt v3 起必填；可兼容本地已有的 pre-v3 报告
  }
  listeningArchetype: {
    name: string
    description: string
    confidence: 'low' | 'medium'
    evidence: EvidenceRef
  }
  dimensions: Array<
    ReportInsight & {
      key: 'exploration_familiarity' | 'focus_variety' | 'collection_rhythm' | 'emotional_texture'
      tendency: string
    }
  >
  moodsAndScenes: ReportInsight[]
  notablePatterns: ReportInsight[]
  personalityReflections: ReportInsight[]
  limitations: string[]
}

interface ReportInsight {
  title: string
  description: string
  confidence: 'low' | 'medium' | 'high'
  evidence: EvidenceRef
}

interface EvidenceRef {
  songIds: string[]
  factKeys: string[]
}
```

校验规则：

- 每个主要章节至少一个 evidence；
- `songIds` 必须是输入 dataset 的子集；
- 百分比、数量和时间范围必须能由本地确定性事实复核；
- 标题、段落、关键词和数组长度设置上限；
- 未知字段拒绝或明确丢弃，不能原样写入数据库；
- 报告始终显示娱乐用途、数据范围和时间精度限制。

### 3.4 任务状态

```text
pending → running → succeeded
                  ↘ failed
                  ↘ cancelled
```

v0.3.x 不实现应用重启后的自动续跑。应用启动时将遗留 `pending/running` 任务标记为 `interrupted`，用户可手工创建重试任务。这样可以避免在用户不知情时重新产生模型费用。

取消语义：

- renderer 发出 cancel 请求；
- main 通过 `AbortController` 取消当前网络请求；
- 已取消任务不能随后保存为成功；
- 如果 provider 已经产生费用，UI 只能说明“已停止等待后续结果”，不能承诺零费用；
- retry 创建新 job，并记录 `retry_of_job_id`。

### 4.2 schema 7：v0.3.0

`llm_profiles`：

| 字段                                  | 用途                                             |
| ------------------------------------- | ------------------------------------------------ |
| `id`                                  | 本地 profile ID                                  |
| `name`                                | 用户可读名称                                     |
| `protocol`                            | 首版实现 `openai_compatible`，通过 registry 扩展 |
| `base_url`                            | 通过 URL policy 的公开地址                       |
| `model`                               | 模型标识                                         |
| `timeout_ms`                          | 请求超时                                         |
| `output_mode`                         | `json_schema` / `json_object` / `prompt_json`    |
| `language`                            | 报告语言，首版默认 `zh-CN`                       |
| `max_input_songs`                     | 输入上限                                         |
| `secret_ref`                          | safeStorage key 引用，不是 API Key               |
| `is_active`                           | 当前活动 profile                                 |
| `last_tested_at` / `last_test_status` | 不含敏感响应的连接测试摘要                       |
| `created_at` / `updated_at`           | 审计时间                                         |

`job_runs`：

| 字段                                        | 用途                 |
| ------------------------------------------- | -------------------- |
| `id` / `kind` / `status`                    | 任务身份与状态       |
| `stage`                                     | 当前阶段             |
| `progress_current` / `progress_total`       | 可选进度             |
| `input_summary_json`                        | 脱敏输入摘要         |
| `error_code` / `safe_message`               | 结构化错误与安全文案 |
| `retry_of_job_id`                           | 重试来源             |
| `created_at` / `started_at` / `finished_at` | 生命周期             |

`ai_disclosure_consents`：

| 字段                         | 用途                           |
| ---------------------------- | ------------------------------ |
| `id` / `profile_id`          | 授权身份与 profile             |
| `target_origin` / `protocol` | 绑定目标 origin 和兼容协议     |
| `source_type` / `source_id`  | 绑定 liked、全部缓存或指定歌单 |
| `fields_hash`                | 绑定已经披露的字段集合         |
| `max_input_songs`            | 允许跳过弹窗的最大输入数量     |
| `created_at` / `updated_at`  | 授权时间                       |

约束：

- secret 实际值继续复用 `SecureStorageService`；
- 删除 profile 时同步删除对应 secret；
- 删除失败时不能留下 renderer 可读的 key；
- `job_runs` 不保存完整 prompt、Authorization header 或 provider 原始错误体。
- 公开设置 `ai_disclosure_confirmation_mode` 支持 `always` 与 `allow_remembered`；
- target origin、protocol、source、fields hash 或输入上限扩大时，旧披露授权失效。

### 4.3 schema 8：v0.3.1

`ai_reports`：报告身份、用户标题、状态、content schema version、provider/model、prompt template version、dataset digest、内容 JSON、生成时间与软删除时间。

`ai_report_sources`：报告与来源快照的独立记录，包括 source type/id/name、dataset schema version、歌曲 ID 快照、时间精度、数量、摘要 JSON 和生成时间。

报告属于用户成果：清理 liked/歌单缓存不删除报告。只有用户在报告历史中明确删除，或执行“彻底删除本地成果”时才能删除。

### 4.4 v0.3.2

优先复用现有 `export_records`，通过 `source_type = ai_report`、`source_id = report_id` 记录导出。只有当版本比较需要独立 revision 时，才新增 `ai_report_generations`；不为导出格式单独建表。

## 5. IPC 与 preload API

新增 API 按领域划分：

```text
llmProfiles:list/create/update/delete/setActive
llmProfiles:setApiKey/deleteApiKey/testConnection
aiDisclosure:preview
jobs:list/get/cancel
aiReports:generate/list/get/rename/delete/regenerate
aiReports:previewExport/export
```

要求：

- create/update 使用运行时 validator，拒绝多余或非法字段；
- `setApiKey` 是唯一接受明文 key 的 IPC，参数只在 main 中短暂存在；
- `list/get` 只返回 `hasApiKey`；
- Base URL 返回给 renderer 前移除 query、fragment 和 userinfo；
- generate 只接受 source、profile ID、输入数量、语言和已确认 disclosure token；
- disclosure token 短时有效、单次使用，并绑定 dataset digest、profile 和目标域名；
- renderer 不能传完整 prompt、任意文件路径或 Authorization header。

## 6. 网络、安全与隐私

### 6.1 URL policy

默认策略：

- 公网 provider 只允许 HTTPS；
- 本地模型允许 `http://localhost`、`http://127.0.0.1` 和 `http://[::1]`；
- 拒绝 `file:`、`ftp:`、data URL、带 userinfo 的 URL 和非 localhost 明文 HTTP；
- 默认拒绝 RFC1918、link-local、loopback 别名和保留地址；
- 请求前解析主机地址，重定向逐跳重新校验；
- 日志仅保存 scheme、hostname、port 和归一化 path，不保存 query 与 key。

### 6.2 网络范围

v0.3.x 不提供应用内代理配置，不新增 HTTP、SOCKS、PAC 或代理凭据设置。网络层只负责目标 URL 安全、协议 adapter、超时、取消、重定向和错误分类。未来只有出现明确产品需求时才单独规划代理能力。

### 6.3 数据披露

每次真实生成前展示并确认：

- 来源名称与类型；
- 歌曲数量；
- 将发送的字段；
- 昵称是否包含；
- provider/profile、model、目标域名；
- 数据处理不可撤回与可能产生费用的说明。

连接测试只发送最小固定文本，不发送音乐数据。生成弹窗允许记住确认；设置页可在 `always` 与 `allow_remembered` 间随时切换并清除历史授权。记住的授权只在 profile、target origin、protocol、source、fields hash 和输入范围仍匹配时有效。

## 7. 错误模型

对 renderer 暴露稳定错误码：

```text
LLM_PROFILE_INVALID
LLM_SECRET_UNAVAILABLE
LLM_AUTH_FAILED
LLM_RATE_LIMITED
LLM_TIMEOUT
LLM_CANCELLED
LLM_NETWORK_ERROR
LLM_ENDPOINT_BLOCKED
LLM_MODEL_NOT_FOUND
LLM_OUTPUT_INVALID
LLM_PROVIDER_ERROR
AI_DATASET_EMPTY
AI_DISCLOSURE_EXPIRED
AI_REPORT_NOT_FOUND
```

原始 provider 错误只在 main 中转换为安全摘要。响应 body、header、key、完整 prompt 和歌曲明细不进入普通日志。

- [ ] 导出失败不损坏报告；
- [ ] HTML 无脚本、无远端请求；
- [ ] 大段文本、空章节和中文文件名布局可用；
- [ ] 打包态 E2E 覆盖报告导出。

## 10. 发布门禁

任一条件不满足不得发布对应稳定版本：

- 远程 CI 或目标架构打包 workflow 失败；
- migration 不能从 schema 6 连续升级；
- renderer、日志、诊断或普通备份可读取 API Key；
- 未经披露确认可以发出音乐数据请求；
- Base URL 可访问被策略禁止的地址或非法重定向；
- 取消任务仍可能保存为成功；
- 非法结构化输出被当作成功报告；
- evidence 引用输入外歌曲或不可复核指标；
- 清缓存会删除报告；
- 导出包含用户关闭的身份字段；
- 国内兼容 provider、localhost provider 或多 profile 切换没有人工记录。

## 11. 风险与控制

| 风险                         | 控制                                                                  |
| ---------------------------- | --------------------------------------------------------------------- |
| OpenAI-compatible 实现差异大 | profile 记录 output mode；adapter contract fixtures；失败时不静默降级 |
| 模型返回非 JSON 或多余字段   | 严格 parse/validate/normalize；失败任务保留安全错误，不保存原始正文   |
| 报告幻觉                     | 本地事实优先、evidence 子集校验、免责声明、禁止诊断式结论             |
| 用户数据或 key 泄露          | main-only transport、safeStorage、披露预览、全链路脱敏、假凭据回归    |
| 开发测试 provider 混入发行版 | development-only env、packaged runtime 拒绝、构建值扫描、空白模板     |
| 自定义 Base URL SSRF         | HTTPS/localhost policy、DNS/IP 检查、redirect 逐跳复核                |
| 取消与响应竞态               | JobManager 原子状态转换；成功写入前再次检查 cancellation              |
| 报告增长占用空间             | v0.3.1 提供历史删除；v0.7 或 v1.0 再完善保留策略                      |
| 真实模型费用不可控           | 默认输入上限、每次披露确认、无自动续跑、连接测试最小请求              |

## 12. 已确认决策

2026-07-21 已确认：

1. 多 LLM profile、单一活动 profile，并提供统一 API 配置与模型管理界面；
2. v0.3.x 不提供应用内代理配置；
3. 生成前显示披露确认，允许记住选择，并在设置中随时调整或撤销；
4. Markdown/HTML 是 v0.3.2 稳定范围，PDF 只有质量达标才加入；
5. 首个真实 provider 只用于本地开发联调，不作为内置服务或 Release 配置；
6. v1.0 的网易云能力完全只读，原 v0.4.x/v0.5.x 整体迁移到 v2.0。

LLM 决策的完整理由、约束和数据模型影响见 `docs/v0.3-decisions.md`；跨版本范围见 `docs/v1-v2-scope-decisions.md`。
