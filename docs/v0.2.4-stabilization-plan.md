# WaveYourYarn v0.2.4 稳定化版本开发计划

> 实施状态（2026-07-17）：S0–S6 已完成。Node 22.22.3 下 `pnpm run check` 通过，包含 8 个测试文件、19 个自动化用例及 main / preload / renderer 生产构建；Electron 开发环境已完成启动级冒烟检查。需要真实网易云账号与网络的登录、同步和实际文件导出仍列为发布前人工回归项。

## 1. 版本定位

WaveYourYarn 当前已经完成：

```text
v0.1.0 工程初始化
v0.1.1 网易云网页登录与登录态管理
v0.1.2 “我喜欢的音乐”同步与本地缓存
v0.1.2.1 歌曲真实顺序与收藏时间修复
v0.1.3 CSV / JSON / Markdown 导出
v0.2.0 用户歌单列表读取与展示
v0.2.1 指定歌单歌曲同步与展示
v0.2.2 指定歌单导出与导出系统扩展
v0.2.3 基础音乐数据统计与 AI 分析数据底座
```

`v0.2.4` 是进入 `v0.3` AI 音乐报告之前的稳定化版本。

本版本不追求新增大功能，重点解决现有数据链路中的一致性、账号隔离、设置安全、数据库升级和回归验证问题，使 `v0.2` 本地音乐数据中心成为可靠的数据底座。

版本完成后再进入：

```text
v0.3.0 LLM 配置与 AI 报告基础架构
```

## 2. 当前基线

当前代码基线：

```text
package version: 0.2.3
branch: main
typecheck: 通过

build: 通过
自动化测试: 尚未建立
```

当前已经具备稳定化所需的主要架构边界：

```text
renderer -> preload -> IPC -> service -> repository -> SQLite
                                  -> adapter -> 网易云 API
                                  -> Electron dialog / shell / fs
```

`v0.2.4` 应继续保持：

1. renderer 不直接访问 Cookie、SQLite、Node.js 或文件系统；
2. preload 只暴露必要且参数受控的 API；
3. main 负责账号、缓存、数据库迁移、文件和外部 API；
4. service 负责事务和业务一致性；
5. repository 负责数据库读写，不在 UI 中拼接 SQL；
6. 所有可破坏本地数据的操作必须明确范围，并可以重新同步恢复。

## 5. 关键技术决策

### 5.1 单活动账号缓存所有权

本版本采用“单活动账号缓存”策略，不在同一个数据库中实现完整多账号分区。

新增内部设置：

```text
cache_owner_ncm_user_id
```

规则：

1. 第一次登录且没有缓存所有者时，将当前网易云用户 ID 记录为缓存所有者；
2. 同一账号退出后，本地缓存可以保留，便于离线查看和重新登录后继续使用；
3. 同一账号重新登录时复用已有缓存；
4. 登录账号与缓存所有者不一致时，在一个事务中清理旧账号的可再同步数据；
5. 账号切换清理范围包括歌曲、歌单、歌单关联、同步状态和导出历史记录；
6. 已经导出的实际文件不删除；
7. 登录结果需要返回 `cacheReset` 或等价的非敏感状态，renderer 明确提示用户旧账号本地缓存已清理；
8. `cache_owner_ncm_user_id` 是 main 内部状态，不允许通过通用 settings API 暴露。

这样可以先保证数据不会串号，同时避免在稳定化版本中进行高风险的多账号数据库重构。

### 5.2 “我喜欢的音乐”歌单身份统一

当前可能同时存在：

```text
网易云真实“我喜欢的音乐”歌单
liked:<userId> 合成歌单
```

本版本统一规则：

1. 优先使用已经同步到本地的真实 liked 歌单；
2. 尚未同步歌单列表时，允许创建 `liked:<userId>` 合成歌单；
3. 后续获取真实 liked 歌单时，将合成歌单关联迁移到真实歌单；
4. 合并后删除重复合成歌单；
5. 统计层不再依赖“扣除重复 liked 歌单”的补偿逻辑；
6. 合并过程必须位于数据库事务内。

### 5.3 缓存删除规则

清空“我喜欢的音乐”缓存时：

1. 只删除 liked 歌单的 `playlist_songs` 关联；
2. 删除 liked 合成歌单或按产品规则保留真实歌单元信息；
3. 只删除已经不被任何其他歌单引用的孤立歌曲；
4. 不删除普通歌单及其歌曲关联；
5. 清理对应同步时间和同步结果；
6. 整个操作必须在一个事务内完成。

清空普通歌单缓存时：

1. 删除非 liked 歌单及其关联；
2. 清理仅由这些歌单引用、且不再被 liked 歌单引用的孤立歌曲；
3. 不影响 liked 歌单关联和同步状态。

### 5.4 设置分为公开设置与内部设置

公开设置第一批只包含：

```text
default_export_directory
```

内部设置包括但不限于：

```text
secret:*
cache_owner_ncm_user_id
liked_songs_last_*
playlists_last_*
playlist_tracks_last_*
all_playlist_tracks_last_*
```

要求：

1. `settings:get`、`settings:set`、`settings:get-all` 必须通过白名单；
2. renderer 无法读取或覆盖任意 key；
3. `settings:get-all` 只返回公开设置；
4. 导出目录通过 Electron 目录选择器设置，renderer 不自行构造任意文件路径；
5. 路径不存在时回退到系统 Downloads；
6. 路径无权限时给出可理解错误，不覆盖原设置。

### 5.5 数据库 migration 成为唯一升级入口

新增 `schema_migrations` 表：

```sql
CREATE TABLE IF NOT EXISTS schema_migrations (
  version INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  applied_at TEXT NOT NULL
);
```

要求：

1. migration 按版本顺序执行；
2. 每个 migration 在事务中执行；
3. 成功后才记录版本；
4. 已执行版本不得重复执行；
5. 新数据库与现有 v0.2.3 数据库都可以升级；
6. 启动时不再同时维护一份不断扩大的硬编码最终 schema 和一套无人执行的 SQL；
7. 对旧数据库使用表结构探测完成 001-005 基线兼容；
8. migration 失败时停止业务 IPC 注册并显示明确启动错误，不能带着半升级数据库继续运行；
9. 迁移测试使用临时数据库，不接触真实用户数据库。

## 9. 数据库与本地数据安全要求

1. 禁止使用 `DELETE FROM songs` 处理单一业务缓存；
2. 删除歌曲前必须确认不存在任何 `playlist_songs` 引用；
3. 涉及多个表的写操作必须使用事务；
4. migration 测试禁止读取或复制真实用户 Cookie；
5. 账号切换只删除可重新同步的本地缓存，不删除用户已经导出的文件；
6. 导出历史清理不删除实际文件；
7. migration 失败不能继续注册依赖新结构的 IPC；
